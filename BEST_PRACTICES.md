## TapSDK 文档编写规范

本文档用于指导 TapSDK 官方文档的编写与维护，目标是在不一次性重构既有文档的前提下，逐步建立**清晰、可演进、低误读成本**的文档体系。

## 1. 文档目标与原则

### 1.1 文档目标

TapSDK 文档的目标是：

- **帮助开发者快速判断是否适用**（能力、平台、限制、前置条件一眼可见）
- **帮助开发者顺利完成接入**（按步骤完成配置与初始化）
- **帮助开发者正确使用功能**（接口契约明确、示例可运行、差异不踩坑）
- **降低误读与技术支持成本**（常见问题/排查/错误码可快速定位）

文档不是功能堆砌说明，而是引导开发者完成**决策与操作**的工具。

### 1.2 核心原则

- **入口清晰**：开发者应明确知道当前阶段应该阅读哪一类文档（概览/开发准备/集成指南/功能/服务端/工具/附录）。
- **职责单一**：每一篇文档只解决一个阶段的问题，避免“指南 + API + FAQ”混写。
- **最小扩展**：新增能力优先按既有目录模板补齐，不破坏顶层结构；未建设内容用“未展开/预留”标注。
- **语义优先**：命名必须准确反映真实适用范围（平台/端/版本/是否仅 Unity/是否仅 PC）。
- **Unity 主线**：Unity 为跨端主入口；PC 特性、限制、启动校验等必须**单独标明**，避免误用。
- **差异显式**：所有跨平台差异（Android/iOS/Unity/PC/C++）必须集中写在“平台差异”中，不分散在正文步骤里。

## 2. 整体目录结构约定

顶层目录职责不可混用，内容必须符合其阶段定位。顶层目录与职责约定如下。

- **概览**：回答“是什么/适不适用/差异与边界”。包含能力概览、平台与 SDK 对照、关键限制与适用场景。
- **更新日志**：回答“变了什么/是否影响我/如何升级”。保持版本号、日期、变更点、影响面、迁移说明的固定格式。
- **开发准备**：回答“接入前需要准备什么”。包含开发者中心配置、通用接入前准备、SDK 版本与升级说明等。
- **集成指南**：回答“怎么接入/怎么跑起来”。按平台/端拆分接入流程（Unity/Android/iOS/PC），并提供初始化与配置、平台差异、常见问题。
- **功能模块（客户端）**：回答“如何使用某个能力”。统一使用模块模板（见第 4 节）。
- **服务端 API**：回答“客户端能力对应的服务端怎么调”。包含鉴权方式、调用规范、接口分组（登录/合规/正版/成就/云存档）。
- **开发工具**：回答“有哪些工具/怎么安装/怎么用/怎么下载”。工具型内容不得混入接入指南正文。
- **附录**：回答“我卡住了怎么排查”。集中放 FAQ、错误码与排查、版本记录等高频检索内容。

## 3. 更新日志写作规范（两篇文章：统一版本线 + C++ 版本线）

更新日志面向**已接入/准备升级**的开发者，必须回答三件事：**变了什么、会不会影响我、我该怎么升级**。它不是提交记录，而是可执行的升级摘要。

### 3.1 版本线与排序约定

- **结构约定**：侧边栏/目录中“更新日志”是一个**分类目录**，目录下包含两篇文章（两条版本线）。
- **目录树（文章层级）**：

```text
更新日志
├─ 更新日志（Unity / Android / iOS）
└─ 更新日志（C++ SDK）
```

- **排序**：两篇文章**各自按时间倒序**（最新在最上）。
- **版本号规则**：
  - **Unity/Android/iOS**：使用**统一版本号**（例如 `v3.4.0`），条目内按平台拆分说明差异。
  - **C++ SDK**：使用**独立版本号**（例如 `C++ v1.2.0`），不得混用到统一版本号下。

### 3.2 每个版本条目的结构（精简版）

每个版本条目必须包含以下字段（没有也要写“无”）：

- **版本号**
- **发布日期**
- **变更明细**：按 **新增（Added）/变更（Changed）/修复（Fixed）** 分类（其他类型用标签内联标注，例如 **[废弃 Deprecated] / [移除 Removed] / [安全 Security]**）

以下字段为**条件必填**（满足条件才写）：

- **升级说明（Upgrade）**：当出现**破坏性变更** / 行为变化 / 配置变更 / 依赖变更 / 需要开发者操作时必须写（用 1-5 条步骤即可）
- **影响说明（Impact）**：当影响不直观或容易误用时必须写（用 1-3 条说明清楚“谁受影响/会怎样”）

### 3.3 破坏性变更的写法（必须明确可操作）

只要出现破坏性变更，必须写清：

- **触发条件**：哪些项目/哪些平台会遇到
- **现象**：错误码/日志关键词/行为变化（尽量可搜索）
- **迁移方式**：改哪个配置/哪个 API/顺序如何调整
- **替代方案**：若有（例如新 API、兼容开关、废弃时间表）

避免使用“优化、调整、重构、完善”等模糊措辞；必须落到用户可验证的行为与操作。

### 3.4 统一版本线条目：平台差异写法

在统一版本号下，平台差异必须**同位呈现**，推荐结构：

- **通用（所有平台）**
- **Unity**
  - **Unity（移动端）**
  - **Unity（PC）**
- **Android**
- **iOS**

原则：差异写在差异处，不要把“仅 Android 的变更”写成“通用变更”。

### 3.5 更新日志模板（可复制）

```md
### vX.Y.Z（YYYY-MM-DD）

#### 新增（Added）
- （无）

#### 变更（Changed）
- （无）

#### 修复（Fixed）
- （无）

#### 升级说明（Upgrade，仅在需要开发者操作时出现）
- 1.

#### 影响说明（Impact，仅在影响不直观/易踩坑时出现）
- 1.
```

```md
### C++ vA.B.C（YYYY-MM-DD）

#### 新增（Added）
- （无）

#### 变更（Changed）
- （无）

#### 修复（Fixed）
- （无）

#### 升级说明（Upgrade，仅在需要开发者操作时出现）
- 1.

#### 影响说明（Impact，仅在影响不直观/易踩坑时出现）
- 1.
```

## 4. 功能模块写作模板（强制统一）

每个客户端功能模块（例如“TapTap 登录”）必须包含以下小节，并按固定顺序编写：

- **功能说明**：业务场景、能力边界、前置条件、是否强依赖某平台/版本。
- **接口说明（按平台）**：Android/iOS/Unity（必要时补充 PC/C++ 说明）；只写契约，不写长篇背景。
- **示例代码（按平台）**：Unity/Android/iOS 同位展示；示例要可复制、可运行、包含必要的初始化与错误处理要点。
- **平台差异**：行为差异、回调时机、权限/合规差异、PC 限制、Unity PC 注意事项、C++ SDK 指引链接等。
- **相关服务端接口**：明确对应的服务端接口能力（如登录状态校验/用户信息查询），并说明何时需要调用。

要求：模块模板中的标题命名与顺序必须一致，便于读者横向对比与团队批量维护。

## 5. 跨平台与 PC 内容标注规范

- **平台信息必须显式写在标题或小节中**：例如“接口说明（Android / iOS / Unity）”“Unity（PC）注意事项”。
- **PC 相关内容必须独立成段/成节**：如启动校验、功能限制说明，不得夹在移动端步骤中。
- **C++ SDK 仅作为独立入口或明确链接**：避免在 Unity/原生文档中写成“默认可用”。

## 6. 命名与信息密度规范

- **标题用“任务语义”而不是“实现语义”**：例如“初始化与配置”“平台差异”“常见问题”优于零散的实现细节标题。
- **列表优先、长段落克制**：能用结构化条目表达的，不写成大段叙述。
- **避免承诺式措辞**：如“全平台一致”“必然成功”等；遇到差异与限制必须写清条件与适用范围。

## 7. 迭代与落地方式（不重构前提）

- **先补模板、再补内容**：旧文档优先对齐目录与模块模板，内容逐步迁移补齐。
- **允许空位但必须标注**：对“未展开/预留”模块保留目录占位，避免结构漂移与入口反复改名。
- **新增内容必须归位**：新增能力先判断属于哪个顶层目录，再按模板扩展，不新增“杂项”入口。
